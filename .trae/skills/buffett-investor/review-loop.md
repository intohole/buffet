# Review Loop - 复盘迭代策略

复盘是投资能力持续进化的核心机制。每次分析后自动执行复盘，沉淀经验，下次分析智能检索相关经验。

## 复盘闭环

```
分析执行 → 输出报告 → 生成复盘文档 → 更新持仓/画像 → 经验沉淀 → 下次智能检索 → 持续进化
   ↑                                                                              |
   └──────────────────────────────────────────────────────────────────────────────┘
```

## 持仓与画像更新协议 (Step 7)

每次分析后，自动检查并更新用户持仓、画像和操作历史文件。

**用户数据文件位于项目根目录 `user-data/`，不在技能目录内。**

### 持仓更新规则

```
触发条件: 用户在对话中明确确认了买卖操作

买入确认关键词: "已买入"/"已建仓"/"已加仓"/"买了"
卖出确认关键词: "已卖出"/"已减仓"/"已清仓"/"卖了"

更新动作:
1. 买入: 在 user-data/user-portfolio.md 当前持仓表添加新行(含止损位/目标价)
2. 卖出: 
   - 减少持股数(部分卖出)
   - 或移除持仓(全部清仓)
   - 添加到已清仓记录(含卖出日期)
   - 更新未卖出股票监控
3. 未确认: 不更新持仓，在分析末尾提醒用户确认操作

同步更新 user-data/trade-history.md:
- 添加操作日志(含序号/日期/操作/标的/数量/价格/金额/逻辑)
- 更新操作统计
- 更新操作行为分析

计算更新:
- 总市值 = Σ(持股数 × 当前价)
- 总成本 = Σ(持股数 × 成本价)
- 总盈亏 = 总市值 - 总成本
- 单股仓位 = 该股市值 / 总市值
- 单板块集中度 = 该板块市值 / 总市值
- 现金比例 = (需用户告知总资金)
```

### 画像更新规则

```
每次分析后检查以下项目，有变化则更新 user-data/user-profile.md:

1. 最近活跃: 更新为当天日期
2. 累计分析次数: +1
3. 历史分析偏好: 使用次数+1，更新最近使用日期和常涉及板块
4. 投资风格画像: 若用户行为与画像不符，更新倾向和置信度
5. 能力圈评估: 若用户分析新行业，扩展能力圈(提升熟悉度)
6. 关注板块: 若分析新板块，添加到关注列表
7. 行为特征: 若发现新的行为模式，补充记录(增加触发次数)
8. 持仓风格分析: 根据最新持仓数据更新
9. 个性化建议: 根据本次分析发现的问题，更新建议
10. 认知成长记录: 若用户学到新知识，添加记录
```

### 操作历史更新规则

```
每次分析后检查以下项目，更新 user-data/trade-history.md:

1. 若用户确认买卖操作 → 添加操作日志
2. 更新操作统计(买入/卖出次数/金额/胜率)
3. 更新操作行为分析(止损执行率/操作频率等)
4. 本次分析给出建议 → 添加到"分析建议执行跟踪"
5. 本次建议添加到"待确认操作"
6. 用户确认/拒绝之前的建议 → 更新确认状态
```

## 智能检索协议 (Step 0)

每次分析前，基于当前场景智能检索相关复盘经验，而非简单读最近一篇。

### 检索流程

```
Step 0.1: 提取当前分析的关键词
  从用户请求中提取:
  - 行业关键词: "银行"/"AI算力"/"电力"/"煤炭"/"半导体"等
  - 策略关键词: "高股息"/"成长"/"价值"/"短线"/"中线"等
  - 标的名称: "中际旭创"/"长江电力"等
  - 分析模式: "热点选股"/"买卖时机"/"持仓诊断"等

Step 0.2: 执行Grep检索
  使用 Grep 工具在 docs/ 目录搜索相关复盘:

  示例1: 用户问"帮我选银行股"
  → Grep pattern: "银行", path: "docs/"
  → Grep pattern: "高股息", path: "docs/"
  → 读取匹配的复盘文档

  示例2: 用户问"中际旭创什么时候买"
  → Grep pattern: "中际旭创", path: "docs/"
  → Grep pattern: "AI算力", path: "docs/"
  → Grep pattern: "光模块", path: "docs/"
  → 读取匹配的复盘文档

  示例3: 用户问"帮我从热点选股"
  → Grep pattern: "热点", path: "docs/"
  → Grep pattern: "选股", path: "docs/"
  → 读取匹配的复盘文档

Step 0.3: 读取匹配的复盘文档
  读取所有匹配的复盘文档(最多3篇)，提取:
  - 历史同类分析中哪些判断正确 → 强化
  - 历史同类分析中哪些判断错误 → 修正
  - 相关选股技巧/避坑经验 → 纳入本次分析
  - 待验证假设的最新状态 → 更新

Step 0.4: 兜底机制
  若关键词检索无结果，读取最近1篇复盘文档作为通用经验参考
  使用 Glob "docs/review-*.md" 获取文件列表，读取最新的1篇
```

## 三层复盘法

### 第一层: 执行复盘 (每次分析后立即)

| 检查项 | 说明 |
|--------|------|
| 数据完整性 | 必需数据点是否全部获取？ |
| 流程完整性 | 工作流检查清单是否全部完成？ |
| 判断一致性 | 5层框架的判断是否自洽？ |
| 输出规范性 | 报告是否按模板输出？ |

### 第二层: 归因复盘 (3-5个交易日后)

| 检查项 | 说明 |
|--------|------|
| 判断准确性 | 评级与实际走势是否一致？ |
| 时机精确度 | 入场价是否接近最优？ |
| 风险识别 | 是否有遗漏的风险？ |
| 机会成本 | 是否有更好的选择？ |

**归因分类**:
- 盈利归因: 能力(方法论正确) vs 运气(意外利好)
- 亏损归因: 执行偏差(未按纪律) vs 认知偏差(分析错误) vs 系统性风险(不可控)

**50笔决策质量评分**:
- 回溯最近50笔分析，每笔按-100至100分打分
- 得分<-30的交易，强制写出3点修正动作
- 将得分绘制成折线图，观察趋势方向

### 第三层: 策略复盘 (每月1次)

| 检查项 | 说明 |
|--------|------|
| 胜率统计 | 近期分析的胜率是多少？ |
| 方法论有效性 | 哪层框架的判断最准确/最不准？ |
| 策略迭代 | 需要调整哪些阈值或规则？ |
| 知识更新 | 是否有新的市场规律需要纳入？ |
| 信号-环境匹配 | 同一信号在不同市场状态下表现差异？ |

## 经验沉淀规则

### 何时沉淀经验

1. **每次分析后**: 记录核心判断和待验证假设
2. **判断被证实时**: 强化该判断逻辑，提高权重
3. **判断被证伪时**: 分析原因，修正判断逻辑
4. **发现新规律时**: 纳入方法论或A股特有知识

### 经验分类

| 类别 | 说明 | 存储位置 |
|------|------|---------|
| 选股技巧 | 新发现的筛选条件/指标 | review-loop.md 经验库 |
| 避坑经验 | 新发现的风险/陷阱 | a-stock-specifics.md |
| 策略优化 | 阈值/规则的调整建议 | master-methods.md |
| 市场规律 | 新发现的市场运行规律 | sector-rotation.md |

## 策略优化协议

当复盘发现系统性问题时，触发策略优化：

| 问题类型 | 优化动作 | 影响文件 |
|---------|---------|---------|
| 护城河评分偏差 | 调整评分标准或权重 | master-methods.md |
| 财务门控过严/过松 | 调整阈值 | master-methods.md |
| 热点评分不准 | 调整4维评分标准 | hot-topic-mining.md |
| 时机判断偏差 | 调整买入信号权重 | master-methods.md |
| A股特有风险遗漏 | 新增红旗检查项 | a-stock-specifics.md |
| 板块轮动不准 | 调整周期映射 | sector-rotation.md |
